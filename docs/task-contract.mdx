## Before Starting

- Read `PRD-NEW.md` for product context.
- Read `docs/README.md` for agent workflow.
- Read the relevant `docs/*.mdx` file before changing a feature.
- For Convex code changes, read `convex/_generated/ai/guidelines.md` first.
- Check `git status --short` and preserve unrelated user changes.

## Implementation Contract

- Keep changes scoped to the requested feature or doc update.
- Prefer existing repo patterns over new abstractions.
- Keep Convex as the backend source of truth.
- Do not reintroduce Turso, Drizzle, SQL migrations, or SQL-first server actions.
- Do not require player login accounts for MVP participation.
- Keep all open play and tournament participants backed by player records.
- Update the living docs when product behavior, architecture, validation, or acceptance criteria changes.

## Testing Contract

- New Convex modules or backend behavior need `convex/*.test.ts` coverage with `convex-test`.
- UI helper/business logic needs focused unit tests.
- Auth route changes need auth-specific tests.
- Run `pnpm test`, `pnpm lint`, and `pnpm build` for feature changes unless the change is documentation-only.
- Documentation-only changes must run at least `pnpm lint` and `pnpm build`.

## Handoff Contract

At the end of a task, report:

- Files changed.
- Behavior changed.
- Tests or checks run.
- Any known follow-up work.
- Any ClickUp or PRD sync needed.
